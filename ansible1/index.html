1. Ansible 설치

  • ‘Community Ansible’을 많이들 쓰지만 시험은 컨테이너 기반의 Red hat Ansible Automatin Platform을 사용한다. 
    dnf -y install ansible-*  # ansible-navigator를 설치하기 위함 
podman login utility.lab.example.com
ansible-navigator images # 실행환경 컨테이너 이미지 다운로드

2. 인벤토리

  • 인벤토리를 통해 관리 대상 호스트를 지정한다.
  • 정적 인벤토리 : 직접 ip, hostname 등록
  • 동적 인벤토리 : 직접 만들지 않고 Inventory Provider에게 질의를 통해 파악한다.
    • ex) vsphere(linux), AD(windows)에 ansible이 물어봐서 호스트를 지정함
  • 대괄호를 통해 인벤토리 그룹을 묶는다. 
    [webservers]
web1.example.com
web2.example.com
192.0.2.42
[db-servers]
db1.example.com
db2.example.com
  • 중첩그룹 : 그룹을 묶은 그룹 ( children keyword ) 
    [usa]
washington1.example.com
washington2.example.com
[canada]
ontario01.example.com
ontario02.example.com
[north-america:children]
canada
usa
  • 범위를 사용한 간소화 
    [usa]
washington[1:2].example.com

[canada]
ontario[01:02].example.com

[korea]
192.168.[4:7].[0:255] # 192.168.4.0~192.168.7.255
  • 인벤토리는 반드시 등록 후 확인하여야 한다.
    • ansible 명령을 사용하는 방식 (구버전) 
      ansible -i inventory(inventory file name) --list-hosts web
    • all 그룹, ungrouped 그룹이라는 기본 그룹이 존재한다.
    • ansible-navigator를 사용하는 방식 
      alias = 'ansible-navigator inventory -m stdout'  # ani alias 만들어 두어야 편하다. 
ani -i inventory(filename) --graph canada
  • ansible- 로 시작하는 여러 명령어가 많이 있었는데( ex. ansible-playbook. ansible-doc, ansible-inventory … ) 그걸 통일하기 위해 ansible-navigator 도입했으나, 실상 또 그 뒤에 한 번더 configre, playbook, inventory 등을 붙여줘야 됨

3. Ansible 구성 파일 관리

ansible.cfg

  • 4군데의 위치에 존재할 수 있으며 어떤 경로의 ansible.cfg를 현재 ansible core에서 참조하고 있는 지는 ansible --version 명령으로 확인 가능
    • 그중 /etc/ansible/ansible.cfg -> 제일 낮은 우선순위
    • touch ~/.ansible.cfg 를 한 이후에 ansible --version를 해보면 config file을 ~/ 경로에서 찾음을 알 수 있음
    • touch ansible.cfg 를 현재 디렉토리에서 한 이후에 ansible --version을 해보면 ansible.cfg를 현재 디렉토리에서 찾음을 알 수 있음
    • 환경 변수 설정이 가장 우선순위가 높다 ( ex. ANSIBLE_CONFIG=/tmp/ansible.cfg && export ANSIBLE_CONFIG )
  • 기본적으로 작성할 수 있어야 한다. 아래는 예시이다. 
    [defaults]
inventory = ./inventory  # 이 설정을 해주게 되면 이제 ani -i inventory 와 같이 직접 inventory 파일 설정 안해줘도 됨 
remote_user = user # ssh 일반 user 
ask_pass = false # 시험은 true 라고 해야 함. false는 공개키 설정을 했을 때만 가능 

[privilege_escalation]
become = true # 권한상승 여부
become_method = sudo # sudo로 권한상승 할 것임
become_user = root 
become_ask_pass = false # sudoers 파일에 NOPASSWD 설정이 wheel group에 되어 있을 때만 false

4. Playbook

---
- name: playbook 전체 에 대한 간단한 설명
  hosts: inventory에 등록한 그룹, host명
  tasks:
    - name: 작업(모듈) 1에 대한 설명
      모듈명(FQDN형식으로 적어야함): 
        argument1: 값
        argument2: 값 
    
    - name: 작업(모듈) 2에 대한 설명
      모듈명: 
        argument1: 값
        argument2: 값
  • ansible-navigator doc -m stdout dnf -> /EXAMPLE 와 같이 모듈의 사용법 알아내기
  • playbook을 실행하기 전 문제가 없는지 테스트를 해야 한다.
    • 구문 체크 : ansible-navigator run -m stdout webserver.yml --syntax-check or ansible-playbook --syntax-check webserver.yaml
    • -C : ansible-navigator run -C -m stdout last.yaml
  • 멱등성이 있기 때문에 playbook은 1번만 실행하지 않고 2번 이상 실행을 통해서 changed의 상태 값을 체크한다.

다중 플레이북

  • service 모듈
    • 서비스의 중지 기동과 관련된 모듈
    • list사용이 안되기 때문에 반복문을 사용하여 작성한다.
---
- name: 인트라넷 서비스 구성하기 
  hosts: servera.lab.example.com
  become: true
  tasks:
    - name: httpd firewalld 설치 
      ansible.builtin.dnf:
        name:
          - httpd
          - firewalld
        state: latest
    - name: Test html 페이지 설정
      ansible.builtin.copy:
        content: "Welcome to the example.com intranet!\n"
        dest: /var/www/html/index.html
    - name: Firewall 실행
      ansible.builtin.service:
        name: firewalld
        enabled: true
        state: started
    - name: Firewall http 설정 
      ansible.posix.firewalld:
        service: http
        permanent: true
        state: enabled
        immediate: yes
    - name: httpd 실행 
      ansible.builtin.service:
        name: httpd
        enabled: true
        state: started
    - name: 웹서버 구성 test 
      hosts: localhost
      become: false
      tasks:
    - name: Connect to intranet web server
      ansible.builtin.uri:
        url: http://servera.lab.example.com
        return_content: yes
        status_code: 200

5. 변수 및 팩트

  • 변수의 우선 순위 gruop_vars < host_vars < playbook 선언(직접 < 파일) < CLI 명령 이다.

  • group_vars, host_vars를 사용하여 호스트와 그룹 별로 변수를 설정한다.

    group_vars/
├── webservers/      # webservers 그룹용 디렉토리
│   ├── apache.yml
│   └── php.yml
└── dbservers/       # dbservers 그룹용 디렉토리
    ├── mysql.yml
    └── backup.yml

host_vars/
├── web01/           # web01 호스트용 디렉토리
│   ├── main.yml
│   └── ssl.yml
└── db01/            # db01 호스트용 디렉토리
    ├── main.yml
    └── backup.yml

  • 플레이북에서의 변수 사용 : 직접 선언 또는 파일 선언

    - name: vars demo
  hosts: all
  vars:   # 직접 선언
    user: joe
    my_color: black
  
  vars_files: # 파일 선언
    - myvar.yaml

  tasks:
    # This line will read: Creates the user joe
    - name: Creates the user {{ user }}
      user:
        # This line will create the user named Joe
        name: "{{ user }}"

  • 명령어에서의 사용

    an run myvar.yaml -e package=apache

Vault

: Ansible에서 secret으로 변수를 관리할 때 사용하기 위한 개념이다.

ansible-vault create secret.yaml  # 비밀번호 설정
ansible-vault view secret.yaml # 비밀번호 입력 
ansible-vault view --vault-password-file my.key secret.yaml # key 파일로 관리
  • 보통은 vault 키 파일은 ansible.cfg에 경로를 설정한다. 
    [defaults]
inventory = ./inventory
remote_user = devops
vault_password_file = ./my.key
  • vault를 활용한 user 생성 playbook
    • node의 /etc/login.defs의 ENCRYPT_METHOD의 알고리즘이 무엇인지 확인 후 password에 알고리즘을 설정해주어야 한다.
    ---
- name: create user by ansible vault
  hosts: serverd.lab.example.com
  vars_files:
    - secret.yaml 
  tasks:
    - name: create new user
      ansible.builtin.user:
        name: "{{id}}"
        group: wheel
        password: "{{ password | password_hash('sha512') }}"

팩트

: 관리 대상 node의 시스템 정보를 자동으로 수집하는 변수

  • fact 확인하기

    ---
- name: collect fact
  hosts: serverd.lab.example.com
  tasks:
    - name: show me fact
      debug:
        var: ansible_facts
    ---
- name: show some facts 
  hosts: all
  tasks:
  - name: Prints various Ansible facts
    debug:
      msg: > # '>'이면 미개행(한줄처리)  '|' 이면 여러 줄 쓰면 여러줄 대로 처리 
        The default IPv4 address of {{ ansible_facts.fqdn }}
        is {{ ansible_facts.default_ipv4.address }}

  • 많이 활용되는 Facts

    # 1. OS 배포판 및 버전 확인
# 예: Ubuntu 20.04, RHEL 8.4
ansible_facts['distribution']         # Ubuntu
ansible_facts['distribution_version'] # 20.04

# 2. 호스트명 확인
# 시스템의 현재 호스트명
ansible_facts['hostname']            # webserver01

# 3. IP 주소 확인
# 시스템의 기본 IPv4 주소
ansible_facts['default_ipv4']['address']  # 192.168.1.100

# 4. 메모리 정보 확인
# 전체 및 가용 메모리 (MB)
ansible_facts['memtotal_mb']             # 16384
ansible_facts['memfree_mb']              # 8192

# 5. CPU 정보
# CPU 개수 및 코어 수
ansible_facts['processor_count']         # 2
ansible_facts['processor_cores']         # 4

# 6. OS 계열
# RedHat/Debian 계열 구분
ansible_facts['os_family']              # RedHat

# 7. 디스크 마운트 정보
# 루트 파티션 정보
ansible_facts['mounts'][0]['mount']      # "/"
ansible_facts['mounts'][0]['size_total'] # 107374182400

# 8. 모든 네트워크 인터페이스
# 시스템의 네트워크 인터페이스 목록
ansible_facts['interfaces']              # ["eth0", "lo"]

# 9. 커널 버전
# 현재 실행 중인 커널 버전
ansible_facts['kernel']                  # "4.18.0-305.el8.x86_64"

# 10. 시스템 아키텍처
# x86_64, aarch64 등
ansible_facts['architecture']            # "x86_64"

  • Facts의 활용 예시

    • OS 별 패키지 설치 
      - name: OS별 웹 서버 설치
  hosts: webservers
  tasks:
    - name: RHEL/CentOS에 Apache 설치
      yum:
        name: httpd
        state: present
      when: ansible_facts['os_family'] == "RedHat"
    - name: Ubuntu에 Apache 설치
      apt:
        name: apache2
        state: present
      when: ansible_facts['os_family'] == "Debian"
    • 메모리에 따른 Heap 메모리 설정 
      - name: Java 설정
  hosts: application_servers
  tasks:
    - name: Java 힙 크기 설정
      template:
        src: java.conf.j2
        dest: /etc/java/java.conf
      vars:
        heap_size: "{{ (ansible_facts['memtotal_mb'] * 0.6) | round | int }}"
    • CPU 코어 수에 따른 웹 서버 설정 
      - name: Nginx 워커 프로세스 설정
  hosts: webservers
  tasks:
    - name: Nginx 설정 파일 생성
      template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      vars:
        worker_processes: "{{ ansible_facts['processor_cores'] }}"
        worker_connections: "{{ ansible_facts['processor_cores'] * 1024 }}"

  • fact를 playbook에서 사용하지 않는 다면 fact수집을 비활성화 할 수 있다. playbook 안에서 gather_facts: no를 설정 해주면 됨.

6. 작업 제어

반복문

: 앞서 service 모듈에 대해서는 list를 쓰는게 안되서 task를 각각 생성해줘야 했다. 이런 경우 반복문이 필요하다. 반복문을 사용할 때는 **loop 키워드를 각 task 별로 지정**해주게 되고, 보통은 loop에 쓰이는 배열을 아래와 같이 변수로 설정하여서 사용한다. 꼭 playbook 내가 아닌 group_vars, host_vars에서 설정해도 됨.

# variables.yaml
---
my_services:
  - postfix
  - dovecot
  - httpd

# playbook
---
vars_files:
  - vars/variables.yaml

tasks:
  - name: services are running
    ansible.builtin.service:
      name: "{{ item }}"
      state: started
    loop: "{{ my_services }}"

조건문 ( when )

---
- name: Simple Boolean Task Demo
  hosts: all
  vars:
    run_my_task: true
  tasks:
    - name: httpd package is installed
      ansible.builtin.dnf:
        name: httpd
      when: run_my_task
  • 조건문을 위한 연산자
    • <, >, <=, >=, !=, is defined, is not defined, not, in, and, or
    • in의 예시 조건문 
      ---
- name: Demonstrate the "in" keyword
  hosts: all
  gather_facts: yes
  vars:
    supported_distros:
      - RedHat
      - Fedora
  tasks:
    - name: Install httpd using dnf, where supported
      ansible.builtin.dnf:
        name: http
        state: present
      when: ansible_facts['distribution'] in supported_distros and ansible_facts['architecture'] == "x86_64"

2가지 이상의 조건들 처리

# and 조건만 여러 개일 때는 - 로 나열할 수 있음
when:
  - ansible_facts['distribution_version'] == "9.0"
  - ansible_facts['kernel'] == "5.14.0-70.13.1.el9_0.x86_64"

# 2개 이상의 and 조건을 or로 묶을 때는 ( ) 사용
when: >
  ( ansible_facts['distribution'] == "RedHat" and
    ansible_facts['distribution_major_version'] == "9" )
  or
  ( ansible_facts['distribution'] == "Fedora" and
  ansible_facts['distribution_major_version'] == "34" )
  • 패키지 그룹으로 설치 (디스크 용량 고려) 
    ---
- name: example 
  hosts: database_dev
  tasks:
    - name: when test 
      dnf:
        name: '@Container Management'
        state: present
      when:
        - ansible_facts.mounts.0.mount == "/"
        - ansible_facts['mount'][0]['size_available'] >= 1000000000

Register 변수를 활용한 분기

: task의 실행 결과를 변수에 저장하기 위함. 그리고 그 결과에 따라 playbook 내에서 분기하기 위하여 사용됨

---
- name: register demo
  hosts: localhost
  gather_facts: false 
  tasks:
    - name: using command module
      command: date 
      register: aaa

    - name: print aaa value
      debug:
        var: aaa
  • register 변수를 위와 같이 출력했을 때, 아래와 같이 나오며 보통 결과의 성공 여부를 판단할 때 rc값을 기준으로 한다. 
    TASK [print aaa value] **********************************************
ok: [localhost] => {
    "aaa": {
        "changed": true,
        "msg": "All items completed",
        "results": [
          {
          "ansible_loop_var": "item",
          "changed": true,
          "cmd": "echo This is my item: one", "delta": "0:00:00.004519",
          "end": "2022-06-29 17:32:54.065165", "failed": false,
          ...output omitted...
          "item": "one",
          "msg": "",
          "rc": 0,
          "start": "2022-06-29 17:32:54.060646", "stderr": "",
          "stderr_lines": [],
          "stdout": "This is my item: one", "stdout_lines": [
                              "This is my item: one"
                          ]
          }
    # register를 통한 분기 예시
---
- name: Restart HTTPD if Postfix is Running
  hosts: all
  tasks:
    - name: Get Postfix server status
      ansible.builtin.command: /usr/bin/systemctl is-active postfix
      register: result
    - name: Restart Apache HTTPD based on Postfix status
      ansible.builtin.service:
        name: httpd
        state: restarted
      when: result.rc == 0

조건문과 반복문의 결합

: 사용 가능한 공간이 300MiB 이상인 /에 마운트된 파일 시스템이 있는 경우, ansible.builtin.dnf 모듈에서 mariadb-server 패키지를 설치하는 예시

- name: install mariadb-server if enough space on root
  ansible.builtin.dnf:
    name: mariadb-server
    state: latest
  loop: "{{ ansible_facts['mounts'] }}"
  when: item['mount'] == "/" and item['size_available'] > 300000000

핸들러

  • 다른 작업에서 트리거한 알림에 응답하는 작업이다.
  • 일반적으로 주요 설정이 변경됐을 때 핸들러 호출을 통해 node 재부팅 및 서비스 재시작에 사용된다.
  • tasks가 모두 끝난 뒤에 핸들러의 작업이 실행된다.
  • handlers 사용 예시 
      tasks:
    - name: copy demo.example.conf configuration template
      ansible.builtin.template:
        src: /var/lib/templates/demo.example.conf.template
        dest: /etc/httpd/conf.d/demo.example.conf
      notify:
        - restart mysql
        - restart apache
  handlers:
    - name: restart mysql
      ansible.builtin.service:
        name: mariadb
        state: restarted
    - name: restart apache
      ansible.builtin.service:
        name: httpd
        state: restarted
  • handler는 호출 순이 아니라 handlers에 선언된 순서로 실행된다.

작업 오류 처리

작업의 실패하는 것에 대해서 어떻게 제어할 것인가.

  • 작업 실패 무시

    - name: Latest version of notapkg is installed
ansible.builtin.dnf:
  name: notapkg
  state: latest
ignore_errors: yes

  • 작업 실행 후 핸들러 강제 실행

    • 아래 예시에서 notapkg의 설치는 존재하지 않는 패키지이므로 실패하여서, 원래라면 notify handler task로 인한 handler가 실행되지 않아야 하지만, force_handlers:yes를 설정함으로써 restart db 핸들러가 실행되게 된다.
     ---
- hosts: all
  force_handlers: yes
  tasks:
    - name: notify handler
      ansible.builtin.command: /bin/true
      notify: restart db
    - name: a task which fails because the package doesn't exist
      ansible.builtin.dnf:
        name: notapkg
        state: latest

  handlers:
    - name: restart db
      ansible.builtin.service:
        name: mariadb
        state: restarted

  • failed_when

    • 사용자가 직접 어떤 경우에 실패인지를 관리하기 위한 구문
      tasks:
    - name: Run user creation script
      ansible.builtin.shell: /usr/local/bin/create_users.sh
      register: command_result
      failed_when: "'Password missing' in command_result.stdout"

  • fail 모듈

      tasks:
    - name: Run user creation script
      ansible.builtin.shell: /usr/local/bin/create_users.sh
      register: command_result
      ignore_errors: yes
    - name: Report script failure
      ansible.builtin.fail:
        msg: "The password is missing in the output"
      when: "'Password missing' in command_result.stdout"

  • changed_when

    • changed_when 구문은 명시적으로 task가 change되었음을 표시하기 위함
    • 실제 조치가 필요한 변화가 node에서 있는 경우, 이를 추적하기 위한 중요한 도구 
        tasks:
    - ansible.builtin.shell:
        cmd: /usr/local/bin/upgrade-database
      register: command_result
      changed_when: "'Success' in command_result.stdout"
      notify:
        - restart_database
  handlers:
    - name: restart_database
      ansible.builtin.service:
        name: mariadb
        state: restarted

  • block 구문

    • block내에 task 여러개를 하나의 task로 묶을 수 있고, 따라서 이를 묶어서 분기 처리 등을 할 수 있다. ( block문에 when을 적용 )
      name: block example
  hosts: all
  tasks:
    - name: installing and configuring DNF versionlock plugin
      block:
      - name: package needed by dnf
        ansible.builtin.dnf:
          name: python3-dnf-plugin-versionlock
          state: present
      - name: lock version of tzdata
        ansible.builtin.lineinfile:
          dest: /etc/yum/pluginconf.d/versionlock.list
          line: tzdata-2016j-1
          state: present
      when: ansible_distribution == "RedHat"
    • rescue, always와 함께 block문 사용하기
      • rescue는 block문이 실패했을 때 실행되고, always는 항상 실행된다.
      tasks:
  - name: Upgrade DB
    block:
      - name: upgrade the database
        ansible.builtin.shell:
          cmd: /usr/local/lib/upgrade-database
    rescue:
      - name: revert the database upgrade
        ansible.builtin.shell:
          cmd: /usr/local/lib/revert-database
    always:
      - name: always restart the database
        ansible.builtin.service:
          name: mariadb
          state: restarted
