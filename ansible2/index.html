1. 파일 배포

파일 배포와 관련된 모듈들.

  • blockinfile: 여러 줄의 텍스트 블록을 삽입, 업데이트 또는 제거 
    ---
- name: file module demo1
  hosts: servera.lab.example.com 
  gather_facts: false 
  tasks:
    - name: using blockinfile
      blockinfile:
        path: /tmp/3day/lunch.txt
        block: |
          10.1.1.1  www.google.co.kr
          10.1.1.2  www.naver.com
  • copy
  • fetch 
    - name: file module demo2
  hosts: all
  gather_facts: false 
  tasks:
    - name: using fetch module
      fetch:
        src: /etc/ssh/sshd_config
        dest: backup
    • 위 플레이북을 실행 후 tree backup을 해보면 backup 디렉토리가 제어 노드에 생성되고 그 하위에 노드별로 디렉토리가 생성되면서 fetch가 됨을 알 수 있다. 즉 src가 관리 node, dest가 제어 node임
  • file 
    ---
- name: file module demo3
  hosts: servera.lab.example.com
  gather_facts: false
  tasks:
    - name: create Dir
      file:
        path: /tmp/3day
        state: directory
        mode: '0760'
    - name: create empty file
      file:
        path: /tmp/3day/lunch.txt
        state: touch
        owner: student
        group: wheel
        mode: 0606
  • lineinfile 
    ---
- name: file module demo 4
  hosts: servera.lab.example.com
  gather_facts: false
  tasks:
    - name: lineinfile test1   # replace
      lineinfile:
        path: /tmp/3day/lunch.txt 
        regexp: '^10.1.1.1'
        line: 10.1.1.4 www.hyundai.com

    - name: lineinfile test 2 # 추가 
      lineinfile:
        path: /tmp/3day/lunch.txt
        line: 59.29.224.9 www.hpe.co.kr
        satate: present
  • stat

Jinja2 템플릿

jinja2 템플릿을 사용하여 관리 node에 hosts파일 셋팅하기

  • network_data 변수 파일 설정 
    network_data:
  - sname: servera
    lname: servera.lab.example.com
    ipv4: 172.25.250.10
    subnet: 255.255.255.0
    gw4: 172.25.250.254
    dns: 8.8.8.8

  - sname: serverb
    lname: serverb.lab.example.com
    ipv4: 172.25.250.20
    subnet: 255.255.255.0
    gw4: 172.25.250.1
    dns: 8.8.8.8
  • jinja2 template 작성 
    # host.j2 template file 작성
{% for host in network_data %}
  {{ host.ipv4 }} {{ host.lname }}  {{ host.sname }}
{% endfor %}
  • template 활용한 playbook 작성 
    ---
- name: using jinja2
  hosts: all
  vars_files:
    - network_vars.yaml

  tasks:
    - name: create /etc/hosts
    - template: 
        src:  ./host.j2
        dest: /etc/hosts

Jinja의 실제 사용 예시

  • Jinja는 보통 미들웨어(nginx, redis, db, kafka, LB, )의 conf파일과 같은 것들을 변수/팩트에 따라 동적으로 관리하기 위해서 사용된다.

Jinja를 쓰지 않으면 정적으로 파일을 copy하거나 blockinfile 등을 통해 직접 작성해주어야 한다.

파일 관련 wrapup 예시

---
- name: Configure system
  hosts: all
  remote_user: devops
  become: true
  tasks:
    - name: Configure a custom /etc/motd
      ansible.builtin.template:
        src: templates/motd.j2
        dest: /etc/motd
        owner: root
        group: root
        mode: 0644
        
    - name: Check file exists
      ansible.builtin.stat:
        path: /etc/motd
      register: motd

    - name: Display stat results
      ansible.builtin.debug:
        var: motd

    - name: Copy custom /etc/issue file
      ansible.builtin.copy:
        src: files/issue
        dest: /etc/issue
        owner: root
        group: root
        mode: 0644

    - name: Ensure /etc/issue.net is a symlink to /etc/issue
      ansible.builtin.file:
        src: /etc/issue
        dest: /etc/issue.net
        state: link
        owner: root
        group: root
        force: yes

2. 복잡한 플레이 및 플레이북 관리

인벤토리 호스트 패턴

  • 명령어에서는 아래와 같이 hosts를 다양하게 지정할 수 있음

    ansible -i inventory1 --list-hosts '*.example.com, !*.lab.example.com'  
ansible -i inventory1 --list-hosts 'db, &london' # db, london 그룹의 교집합에 해당하는 host

  • 플레이북에서는 아래와 같이 hosts를 다중 지정할 수 있음

    ---
- name: host example
  hosts:
    - db
    - lb
    - db, &london

파일 포함하기, 가져오기

  • import_playbook

    name: Play 1
hosts: localhost
tasks:
  - name : task1
    debug:
      msg: Play 1

  - name: Import Playbook
    import_playbook: play2.yml

  • import_tasks : task를 playbook에 정적으로 삽입함

    • import_tasks 사용시 when과 같은 조건문을 같이 쓰게 되면 조건문이 가져온 모든 tasks에 각각 적용됨
    # main.yml
---
- name: Configure system
  import_tasks: ubuntu.yml
  when: ansible_distribution == 'Ubuntu'

- name: Configure system
  import_tasks: centos.yml
  when: ansible_distribution == 'CentOS'

# ubuntu.yml
---
- name: Install nginx
  apt:
    name: nginx
    state: present

- name: Start nginx
  service:
    name: nginx
    state: started

  • include_tasks : task를 playbook에 동적으로(런타임에) 삽입함

    • include_tasks 사용 시에 when을 함께 쓰게 되면 포함을 시킬지 말지에 대한 용도로서 사용되며, 보통은 when문과 함께 사용하여 조건에 맞는 경우만 가져오도록 함.
    # main.yml
---
- name: Include OS specific tasks
  include_tasks: "setup_{{ ansible_facts.ansible_distribution | lower }}.yml"
  when: ansible_distribution in ['Ubuntu', 'CentOS']

# setup_ubuntu.yml
---
- name: Install packages
  apt:
    name: "{{ item }}"
    state: present
  loop:
    - nginx
    - python3

3. 역할

  • 콘텐츠를 그룹화하여 코드를 다른 사용자와 쉽게 공유할 수 있게 해주며 재사용성을 높여줌

  • import_rule, include_rule 은 import_task, include_task와 문법의 거의 비슷함(조건문이 의미하는 방식 등)

  • roles section

    ---
- name: A play that only has roles
  hosts: remote.example.com
  roles:
    - role: role1
    - role: role2

  • role의 디렉토리 구조 ( ex. my-role )

    ├── defaults
│ └── main.yml
├── files
├── handlers
│   └── main.yml
├── meta
│   └── main.yml
├── README.md
├── tasks
│   └── main.yml
├── templates
├── tests
│   ├── inventory
│   └── test.yml
└── vars
└── main.yml
    • defaults: main.yml 파일에 role이 사용될 때 덮어쓸 수 있는 role 변수의 기본값이 포함되어 있습니다. 해당 변수는 우선 순위가 낮으며 플레이에서 변경하고 사용자 지정할 수 있음
    • files: 이 디렉터리에는 role에서 참조한 정적 파일 존재
    • handlers: main.yml 파일에 역할의 핸들러가 정의됨
    • meta: main.yml 파일에는 작성자, 라이센스, 플랫폼 및 옵션 role 종속성 을 포함한 role에 대한 정보가 정의됨
    • tasks: main.yml 파일에는 role의 task가 정의됨
    • templates: role에서 참조한 Jinja2 템플릿이 정의됨
    • tests: role을 테스트하는 데 사용할 수 있는 인벤토리와 test.yml 플레이북이 포함될 수 있음
    • vars: main.yml 파일은 role의 변수 값을 정의합니다. 종종 이러한 변수 는 role 내에서 내부 목적으로 사용되며 우선 순위가 매우 높음. 따라서 플레이에서 사용될 때 변경되지 않음.

  • roles 찾는 위치와 순서

      1. ./roles
      1. /usr/share/ansible/roles
      1. /etc/ansible/roles

  • 특수 작업 세션 : pre_tasks , post_tasks

    • role의 실행 전후에 해야 할 작업을 기술하기 위함. 그러나 실제로는 잘 사용되지는 않고 여러 플레이로 나누거나 role 내부적으로 관리하는 방식이 권장된다.
    • 그런데 roles 섹션으로 role을 가져올 경우 role은 해당 play에 정의된 task보다 먼저 실행된다. roles, tasks의 정의 순서와 무관함. 따라서 아래 playbook에서 작업의 순서는 pre_tasks -> roles -> tasks -> post_tasks 이다.
    - name: Play to illustrate order of execution
  hosts: remote.example.com
  pre_tasks:
    - name: This task runs first
      ansible.builtin.debug:
        msg: This task is in pre_tasks
      notify: my handler
      changed_when: true
  roles:
    - role: role1
  tasks:
    - name: This task runs after the roles
      ansible.builtin.debug:
        msg: This task is in tasks
      notify: my handler
      changed_when: true
  post_tasks:
    - name: This task runs last
      ansible.builtin.debug:
        msg: This task is in post_tasks
      notify: my handler
      changed_when: true
  handlers:
    - name: my handler
      ansible.builtin.debug:
        msg: Running my handler

role 실습

ansible-galaxy role --help
ansible-galaxy role list
ansible-galaxy role init httpd-role
ansible-galaxy role init firewall-role

distribution을 만족하면 apache의 설치 및 초기 설정을 하는 role과 방화벽 설정하는 role을 2가지 쓰는 시나리오 실습

httpd-role

  • task 작성하기 
    # httpd-role/tasks/main.yml
- name: install web pkg
  dnf:
    name: "{{ item }}"
    state: latest
  loop: "{{ httpd_packages }}"
  when: ansible_facts.distribution in supported_dist

- name: copy webpage
  copy:
    src: "{{ src_file_path }}"
    dest: "{{ dest_file_path }}"
  notify: 
    - restart_service
  • files에 index.html 작성하기 
    echo Hello role! > ./files/index.html
  • vars에 변수 선언하기 (task에 작성했던 변수) 
    # httpd-role/vars/main.yml
supported_dist:
  - CentOS
  - RedHat
httpd_packages:   # dnf search httpd 
  - httpd
  - httpd-manual
  - mod_ssl
src_file_path: ../files/index.html
dest_file_path: /var/www/html/
service_name: httpd # handler에서 사용
  • handlers 작성 
    # httpd-role/handlers/main.yml
- name: restart_service
  service: 
    name: "{{ service_name }}"
    state: restarted

firewall-role

  • task 작성 
    - name: add rule to firewalld 
  firewalld:
    service: "{{ item }}"
    permanent: true
    immediate: true
    state: enabled 
  loop: "{{ service_port }}"
  • vars 작성 
    service_port:
  - http
  - https

Playbook에서 role을 사용하는 방법

  • task로 role import/include하는 방법

    - name: httpd-role demo
  hosts: servera.lab.example.com
  tasks:
    - name: dummy msg
      debug:
        msg: "let's start role" 
    - name: import httpd-role
      import_role:
        name: httpd-role

  • roles 섹션 사용하는 방법

    - name: firewall-role demo
  hosts: serverb.lab.example.com

  roles:
    - httpd-role
    - firewall-role
  
  tasks:
    - name: print dummy msg
      debug:
        msg: "Finished roles"

실제로는 어떻게 role을 사용할 것인가?

  • 간단하게는 roles 사용
  • 조건부 실행이 필요하거나 동적 실행이 필요하다면 roles쓰지 않고 import_role, include_role사용
    • import_roles: loop와 함께 사용할 수 없으며 조건부로 포함시키고 싶을 때 사용 
      - name: Setup web servers
  hosts: web
  tasks:
    # 조건부 실행 예시
    - name: Import nginx role
      import_role:
        name: nginx
      when: enable_web_server | bool    # role 전체에 조건 적용
      tags: ['web', 'nginx']           # role 전체에 태그 적용
      vars:
        nginx_port: 8080
    • include_rules: loop 사용 및 동적 실행 
      - name: Setup applications
  hosts: app_servers
  tasks:
    # 동적 실행 예시
    - name: Include service roles
      include_role:
        name: "{{ item }}"
      loop:                           
        - redis
        - mongodb
        - rabbitmq
        
    # 동적 role 이름 예시
    - name: Include environment specific role
      include_role:
        name: "{{ env }}_config"      # 변수로 role 이름 지정
      vars:
        env: "{{ lookup('env', 'DEPLOY_ENV') }}"

4. 컬렉션

  • role을 다운 받는 것은 일반적이지 않고 컬렉션을 받는게 더 일반적임.
  • role, module, lib 등이 컬렉션 안에 포함이 되어 있음
  • 삭제는 직접 rm 해줘야 함
ansible-galaxy collection --help
ansible-galaxy collection download openstack.cloud -p ./collections  # tar.gz download 후 install 가능
ansible-galaxy collection install openstack.cloud:{version} -p ./collections # install
ansible-galaxy collection list

5. Ansible 문제 해결

플레이북 문제해결

  • alias 사용하기 ( an )
  • vi 에디터 환경설정
  • playbook 실행 전에 syntax 확인하기
  • register 키워드를 통해 실행결과를 변수에 저장하고, 그 변수의 내용을 확인하기 ( debug var 활용. rc, stdout 확인 )
  • playbook 아티팩트(실행로그) 참조
    • 아티팩트를 사용하지 않으려면 ~/.ansible-navigator.yml 설정을 아래와 같이 변경 
      ansible-navigator:
  playbook-artifact:
    enable: false
  • ansible.cfg 에 log파일을 별도 설정 
    [defaults]
...
log_path = ./lasttime.log

관리 호스트 문제해결

대부분 인증/권한과 관련된 문제이다.

  1. 일반 유저로 ssh 접속이 되는지
  2. 유저의 인증을 패스워드 인증이 아닌 공개키 인증 설정으로 해야 함
  • 그러면 최초에 공개키를 관리 node로부터 받으려면 어떻게 해야하나?
  • ansible.cfg에 파일에 ssh 연결에 대한 설정 필요 
    [defaults]
...
host_key_checking = false
  1. sudo 를 통해 권한상승 시 암호 입력 받지 않도록 설정 필요 ( /etc/sudoers )
